1. Introduction

This document compiles various concrete use cases implementing the suggested <standOff> element in order to a) identify possible annotation configurations that may be used as recommended practices for the corresponding phenomena and b) refine the possible constraints that may bear upon it and its sub-elements. It is based on the experience gained in various projects since the initial workshop that took place in January 2014 and which launched the definition of this element, as well as on some ongoing activities that are being pursued in several national projects in France and Germany (among others), in international institutions (EPO) and in the content of several ISO projects (some of which have already been published as standards).

The major conclusions that can be drawn from these use cases are the following:

  • There are only two new elements to add to the TEI space:
    • <standOff>: the general container, designed as a mini-‘<TEI>’ document dedicated to gather all annotation related information for a given annotation layer
    • <listAnnotation>: a list like element within <standOff> to allow various types of groupings for stand-off annotations
  • The content model of <annotationBlock> must be revised to integrate the multiple usage scenarios that have been identified. Basically we need an alternative (<choice>) between 2 or 3 possible content models.

2. Overview of the <standOff> proposal

Note: This section should be conceived as text that would go in the guidelines

This proposal implements embedded stand-off annotations, i.e. annotations to an existing document that have the following properties:

  • Annotations do not alter any part of the document itself, header nor content
  • There may be as many complementary and competing annotations attached to a given document as wanted
  • Annotations occur at a precisely determined place in the TEI document organisation, namely as an additional resource, child of <TEI> or <teiCorpus> Note: We need a scenario for such a usage
  • Annotations are conveyed with their specific meta-data, to convey information about origin (software, annotators), specific re-use rights, encoding schemes etc.

2.1. The <standOff> element

The <standOff> element is the container of any group of coherent annotations attached to a TEI document. Its main characteristics are:

  • It is attached to the top level of a TEI document by being a member of model.resourceLike
  • Its content model resembles that of a TEI document by comprising a header (<teiHeader>), additional resources (model.resourceLike) and a content (one or several <listAnnotation> elements)

The actual nature of the <standOff> can be further specified by means of two attributes:

  • type: to indicate the general type of annotations which is being represented
  • notation: to elicit the specific notation used for the corresponding type of annotation in reference to an external standard or a documented practice.

2.2. The <listAnnotation> element

The <listAnnotation> element is a general container allowing one to structure annotations in various subsections. In this respect, its role is analogous to the one played by <div> in a general TEI document. Still, <div> could not be taken up here in order to be able to keep to a very restrained content model.

As an iterative and recursive element, <listAnnotation> can be typed to indicate the organisational logic of the annotations (e.g. per annotation layer, per annotator, per annotation objects in the case of NER).

The <listAnnotation> element is also structured like other listXXX elements so that it may start with model.headLike members (e.g. <head>).

2.3. Updating the <annotationBlock> element

The <annotationBlock> was initially conceived as a construct with a dual role for the inline and stand-off annotation of textual content. For the sake of having it quickly inserted in the TEI guidelines, only a limited range of usages has been documented so far.

In the stand-off annotation proposal, <annotationBlock> takes a major role as the container for elementary annotations.

2.4. Taking up existing annotations mechanisms from the TEI vocabulary

The TEI guidelines has a wide variety of elements to implement elementary mechanisms for stand-off annotation. The purpose of this document is to provide more precise guidance as to how to combine them and identify their complementarities. As this stage we can characterised these elements as follows:

  • <span> is a general purpose element to point to a location or a sequence in a document and ideal to cover the general notion of markable in the linguistic annotation literature.
  • <zone>, which can be seen as playing a similar role to <span> in the case of images
  • <interp>, which currently has a narrow scope in the TEI guidelines but offers the possibilities to connect annotated passages (inst) with annotation content (ana)

3. Aligning with the W3C Web Annotation Data Model (formerly known as Open Annotation model)

3.1. Quick presentation of the Web Annotation Data Model

The Web Annotation Data Model (WADM) articulates around three complementary components:

  • target: that identifies one or several resources about which something is said
  • body: that contains what is said about the resource
  • annotation: that links one or several targets with 0-n bodies

This model decouples resources and what is said about them by means of the annotation component. It is highly flexible and powerful since it allows one to associated as many targets with as many bodies as needed.

It is suggested that the Open Annotation model could be implemented as in the <standOff> construct follows:

  • The Target component could be implement with either <span> for textual targets or <zone> when pointing to an image.

    Note: See the proposed implementation of <annotationBlock>: we suggest to introduce a class model.OATarget

  • The Body component can be implemented by means of a wide variety of elements depending on the various use cases that we describe here.

    Note: See the proposed implementation of <annotationBlock>: we suggest to introduce a class model.OABody

  • The Annotation component is implemented by means of the <interp> element complemented with the three following attributes:

    • type: to indicate the type of the annotation (e.g. ‘quantity’, ‘person’, etc.)
    • inst: points to one or more <span> elements (possibly in the same <annotationBlock> when appropriate
    • ana: points to the actual content of the annotation, depending on the annotation scheme. In simple cases, these could be <note> (see below) or <fs> elements.

    Additional elements could probably be used here, but we should provide univocal guidance for such a use case, in order to avoid content model ambiguities and in general too much heterogeneous implementations of the WADM model.

    Note: See the proposed implementation of <annotationBlock>: we suggest to introduce a class model.OAAnnotation

3.2. Simplest annotation with <note>

The following example shows a simple WADM compliant construct where a span of text is associated to two different notes made by different annotators. The resp attributes would refer here to persons described in the <teiHeader> of the embedding <standOff>.

<annotationBlock>  <note xml:id="n1" resp="#annotator1">This series of sentences .... </note>  <note xml:id="n2" resp="#annotator2">This series of sentences .... </note>  <interp ana="#n1 #n2" inst="#sp1"/>  <span from="#s1" to="#s3" xml:id="sp1"/> </annotationBlock>

3.3. Named entity recognition within the WADM model

3.3.1. Introduction

Named entity recognition is a typical use case for stand-off annotation. It is usually based upon an automatic analysis process of a digital text which produces various hypotheses as to which entities (persons, events, organisations, dates, etc.) are actually mentioned. Given the variety of possible hypotheses, and the quantity of information that a NER module may produce, it does not make sense in putting the resulting annotation inline. Besides, there is a need to record some specific metadata related to the NER process such as the tool that has been used, the parameters given to the tool, etc.

The idea here is to contribute to the definition of a standardised output interface for NER components or online services so that such annotations can actually be re-used, compared, visualised in a systematic way.

3.3.2. General document organisation for NER

<TEI xmlns="http://www.tei-c.org/ns/1.0">  <teiHeader> <!-- The header of the main document being annotated -->  </teiHeader>  <standOff>   <teiHeader> <!-- The header of the annotation part -->   </teiHeader>   <listAnnotation> <!-- The annotations -->   </listAnnotation>  </standOff>  <text>   <body>    <p> <!-- The text being annotated, possibly segmented (e.g. in <w>) -->    </p>   </body>  </text> </TEI>

3.3.3. Grouping NER annotations with <listAnnotation>

NER annotations are usually organised according to the type of recognised entities. This facilitates the further filtering of the output of a NER Process when a user only needs a specific type of entities. A typical output may thus be represented as follows:

<standOff>  <listAnnotation type="individuals"> ... </listAnnotation>  <listAnnotation type="places">...</listAnnotation>  <listAnnotation type="organizations">...</listAnnotation>  <listAnnotation type="dateTimes"/>  <listAnnotation type="events"/> ... </standOff>

4. Morphosyntactic annotation

ISO 24611 (MAF) contains a TEI based serialisation for morpho-syntactic annotations. The underlying model of ISO 24611 is based upon a two level model distinguishing between:

  • Tokens: elementary units segmented in the source textual content, and
  • Word forms: the abstract lexical units that may group together one or several tokens.
Note: A token can also be associated to more than one word forms, e.g. in French: du, combining de and le.

In this use case, we make the assumption that the text is tokenized inline by means of <w> elements as follows, with each token being associated to a unique identifier by means of an xml:id attribute:

<text>  <body> ... <p>    <w xml:id="w1">I</w>    <w xml:id="w2">wanna</w>    <w xml:id="w3">put</w>    <w xml:id="w4">up</w>    <w xml:id="w5">new</w>    <w xml:id="w6">wallpaper</w>    <pc>.</pc>   </p> ... </body> </text>

In conformance to ISO 24611 (MAF), word forms are represented by <span> elements, pointing to the corresponding token(s) in the source text (see the note below concerning the use of the proper attributes). The actual analysis of the word form is provided by means of a feature-structure referred to from the <span> element by means of an ana attribute. The additional corresp attribute allows one to point to the lexical entries equivalent to the word form, when one links the morpho-syntactic annotation to a reference lexicon.

Note: This representation is not conformant to the WADM model. In some way, the <span> element combines both roles of target and annotation classes from WADM. The actual body is made of both the feature structure and the lexical entry that are being pointing to.
Note: To be checked: should we use from and to instead of target here.

5. Reference annotation

Reference annotation consists in identifying referring expressions in text or discourse and qualify the relation between them (e.g. anaphorical links) or with existing or putative external entities (e.g. in the case of deixis).

The work carried out within the project ISO 24617-9 (RAF) distinguishes to this purpose two types of annotation entities between which various types of links can hold:

  • referring expressions: correspond to the actual markables in the text. They may point to text spans, sequences of words (<w>) or higher level objects resulting from other annotation layers (e.g. word forms is the corpus has been morphosyntactically annotated);
  • discourse entities: ...

In the following paragraphs of this section, we consider we have a simple tokenized version of the sentence: ‘Prendre une poire et la couper.’ (Take a pear and cut it.) within a simple TEI document only outlined below:

<TEI xmlns="http://www.tei-c.org/ns/1.0">  <teiHeader>...</teiHeader>  <standOff>...</standOff>  <text>   <body>    <s>     <w xml:id="w0">Prendre</w>     <w xml:id="w1">une</w>     <w xml:id="w2">poire</w>     <w xml:id="w3">et</w>     <w xml:id="w4">la</w>     <w xml:id="w5">couper</w>     <pc>.</pc>    </s>   </body>  </text> </TEI>

In keeping with the ISO 24617-9 model, we organize the annotations in four sub groups marked up by typed <listAnnotation> elements:

<standOff>  <teiHeader>...</teiHeader>  <listAnnotation type="referingExpressions">...<annotationBlock>    <span type="referringExpression"/>   </annotationBlock>...</listAnnotation>  <listAnnotation type="discourseEntities">...<interp type="discourseEntity"/>...</listAnnotation>  <listAnnotation type="lexicalRelations">....<link type="lexicalRelation"/>...</listAnnotation>  <listAnnotation type="objectalRelations">...<link type="objectalRelation"/>...</listAnnotation> </standOff>
Note: Sub-elements can be further structured in <annotationBlock> elements when they correspond to the same textual segment (e.g. sentence, utterance, etc.). This is meaningful for referring expressions, whereas the other annotation objects are more likely to correspond to inter-sentential mechanisms.

For the sentence introduced before,we can markup two referring expressions (‘une poire’, ‘la’), further qualified by means of a feature structure:

<annotationBlock>  <spanGrp type="referringExpression">   <span xml:id="m1" from="#w1" to="#w2"    ana="#fs1"> <!-- une poire -->   </span>   <span xml:id="m2" from="#w4" ana="#fs2"> <!-- la -->   </span>  </spanGrp>  <fs xml:id="fs1">   <f name="syntacticCategory">    <symbol value="nounPhrase"/>   </f>   <f name="determinerType">    <symbol value="indefinite"/>   </f>   <f name="referentialStatus">    <symbol value="discourseNew"/>   </f>  </fs>  <fs xml:id="fs2">   <f name="syntacticCategory">    <symbol value="pronominalPhrase"/>   </f>   <f name="referentialStatus">    <symbol value="discourseOld"/>   </f>  </fs> </annotationBlock>

6. Parallel annotation of spoken utterances

The initial model for <annotationBlock> was conceived to represent the traditional notion of interlinear annotation that one finds, among others, in the field linguistic domain. This representation aligns, in a way resembling a musical score (Partitur), an utterance with various layers of annotations, such as: part of speech, prosody, literal translation etc.

<annotationBlock who="#SPK0" start="#T0"  end="#T2">  <u>   <w xml:id="w1">And</w>   <w xml:id="w2">what</w>   <anchor synch="#T1"/>   <w xml:id="w3">comes</w>  </u>  <spanGrp type="pos">   <span from="#w1" to="#w1">CONJ</span>   <span from="#w2" to="#w2">RELPRO</span>   <span from="#w3" to="#w3">V</span>  </spanGrp>  <spanGrp type="prosody">   <span from="#T0" to="#T1">faster</span>  </spanGrp> </annotationBlock>

In the context of the <standOff> proposal, the idea is to generalise this usage and use the same element for providing the same type of annotation in a stand-off manner by simply pointing to the corresponding <u> from the <annotationBlock> by means of a corresp attribute. This is illustrated in the following example which is equivalent the the previous construct.

<TEI xmlns="http://www.tei-c.org/ns/1.0">  <teiHeader>...</teiHeader>  <standOff type="multilayerTranscription"   notation="ISO24624">   <teiHeader>...</teiHeader>   <listAnnotation>    <annotationBlock corresp="#u1">     <spanGrp type="pos">      <span from="#w1" to="#w1">CONJ</span>      <span from="#w2" to="#w2">RELPRO</span>      <span from="#w3" to="#w3">V</span>     </spanGrp>     <spanGrp type="prosody">      <span from="#T0" to="#T1">faster</span>     </spanGrp>    </annotationBlock>   </listAnnotation>  </standOff>  <text> ... <u xml:id="u1" who="#SPK0" start="#T0"    end="#T2">    <w xml:id="w1">And</w>    <w xml:id="w2">what</w>    <anchor synch="#T1"/>    <w xml:id="w3">comes</w>   </u>  </text> </TEI>

These types of annotations are not specific to the linguistic case and it reflects a general need to associate annotations with streches of text. The following example Note: Courtesy of B. 8. Towards a content model for <annotationBlock>

Note: Intro: focusing on <annotationBlock>

8.1. The baseline

In the initial proposal, the content model of <annotationBlock> was limited to a plain and flat class, model.annotation, with the following constraints added to the elements that one would want to use fro annotaiton purposes (whether as source, annotation or body in the WADM):

<classRef key="model.annotation"  minOccurs="0" maxOccurs="unbounded"/>

The major difficulty with this is that there is no way to constraint that one would want to use specific elements as source, annotation or body.

Suggestion has been made (LB, I think) that such a class should better be called model.annotationBlockPart

8.2. Keeping the continuity with the transcription of speech view

The first constraint bearing upon the content model of <annotationBlock> is to ensure the continuity with the its initial definition and to allow a stand-off verion of interlinear ‘Partitur’ annotation.

<sequence>  <classRef key="model.linguisticSegment"   minOccurs="0" maxOccurs="1"/>  <classRef key="model.interlinearAnnotation"   minOccurs="0" maxOccurs="unbounded"/> </sequence>

8.3. An WADM model compliant content model

In the case of an WADM compliant model, the content model of <annotationBlock> should be based upon three classes:

  • model.OATarget: the class could be pre-filled with such elements as <span> and <surface>.
  • model.OABody
  • model.OAAnnotation: a mandatory class reference for this annotation profile. The class should contain <interp> by default and one should be cautious in adding extra element to avoid ambiguous content models when the added element may also appear in the other classes.
<sequence>  <classRef key="model.OABody" minOccurs="0"   maxOccurs="unbounded"/>  <classRef key="model.OAAnnotation"   minOccurs="1" maxOccurs="1"/>  <classRef key="model.OATarget"   minOccurs="0" maxOccurs="unbounded"/> </sequence>
Note: The naming to be change to be more explicit as to the relation to the WADM W3C proposal: model.WADMBody, etc.

8.4. Combining the various possibilities

In order to differentiate between the different encoding variants we have identified, the <annotationBlock> element must be associated to a notation attribute Note: Could also be mode which can take one of the following values:

  • inSitu: non stand-off usage of <annotationBlock> as documented in the Transcriptions of speech chapter.
  • exSitu: equivalent to the inSitu case in stand-off mode
  • WADM: Open Annotation compliant model (Target, Body, Annotation)
Note: A simplification of the annotation model may consist in adding inst to <annotationBlock>, with the semantics that <annotationBlock> +is+ the annotation level (in the sense of WADM), the target being expressed by the inst and the content of <annotationBlock> is the body level. From an implementation point of view, this would mean adding <annotationBlock> to att.interpLike.
